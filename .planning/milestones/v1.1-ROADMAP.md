## Overview

v1.1 was a hardening + capability-extension milestone bolted onto the shipped v1.0.0 marketplace + zapili plugin. Two phases:

- **Phase 7** closed the five non-blocking follow-ups surfaced by the v1.0.0 ultra-principal code review (C-03 planner prior-findings contract, C-04 orchestrator flagged_gaps routing, F-01/F-02 fixture completion, H-01 check-codex hygiene, S-01 wave-disjointness regex).
- **Phase 8** introduced a NEW capability — a codex self-fix fallback that fires after the validator fix-loop hits its iteration cap with persistent HIGH findings. Instead of halting, the orchestrator dispatches codex with a `fixer` role prompt, codex emits a unified-diff patch against the offending artifact, the orchestrator validates via `git apply --check` and applies it, then re-runs the original validator on the patched artifact. This is the escape hatch for cases where the engineer subagent has reached its own ceiling on a particular issue but codex (independent context, different model family) can resolve it.

Beyond the two planned phases, v1.1 also absorbed three rounds of post-implementation ultra-principal review:

1. **Round 1 (commit `c870186`):** 3 BLOCKERs + 3 WARNINGs from the post-Phase-8 review (B-1 codex-self-fix double-invocation breaking validate=apply guarantee, B-2 f6 fixture false-positive overlap, B-3 schema missing `fix_loop_cap` field, plus 3 warnings on diagnostics + docs).
2. **Round 2 (commit `78a5f30`):** Exhaustiveness contract added to all codex validator prompts to defeat codex's default targeted-review mode. Schema extended with 4 optional fields (`why_real_risk`, `repro`, `tests_to_add`, top-level `not_fully_audited[]`); inline prompts in all three validators gain a verbatim `<exhaustiveness>` XML block.
3. **Round 3 (commit `b31d95a`):** 3 P1 + 5 P2 from the release-prep ultra-principal review (Russian text in production prompts, phase-changes schema max=3 inconsistent with fix_loop_cap=4 default, README.md still showing v1.0.0, finding-ID uniqueness not enforced, $PROMPTS_REF absolute path in prompt heredocs, CALIBRATION.md table stale, SKILL.md hardcoded "3-attempt cap").

## Milestone Summary

**Decimal Phases:** none — both v1.1 phases are integer-numbered.

**Key Decisions:**

- **Phase 7 bundles all five v1.0 review follow-ups** into one atomic-cleanup phase. Rationale: each individual fix is small (1–3 commits) and they share the same intent (close v1.0 review). Bundling avoids the per-phase boilerplate cost without losing per-deliverable traceability via the 3-plan split.
- **Phase 8 dispatches codex against the SPEC (PHASE-XX.md), not the engineer output.** Rationale: the engineer has already tried four times against the current spec and converged on the same answer; the fix must revise the spec. Documented in SKILL.md Stage 7c.1.
- **Codex self-fix is single-attempt per cap-hit.** Rationale: stacking speculative codex patches on top of each other compounds errors. The orchestrator halts after one attempt so a human can inspect; re-running `/zapili:zapili` will fire another self-fix dispatch (Stage 0 preserves the iteration counter from on-disk artifacts).
- **Schema extension is back-compat.** All four new fields (`why_real_risk`, `repro`, `tests_to_add`, `not_fully_audited[]`) are optional — old fixture files validate unchanged. Rationale: avoid breaking existing `expected-findings.json` files in f1..f5.
- **Exhaustiveness contract framing is `(exhaustive coverage, not a targeted re-review)`** — the Round 3 review forced removal of the original Russian phrase per CLAUDE.md's "no Russian in code/tests" rule. The English replacement is calibrated against live codex 0.133.0 to produce the same exhaustive coverage.
